根据用户描述的产品需求，生成面向程序员的技术方案文档。文档结构分为【必要章节】和【可选章节】，必要章节始终包含，可选章节根据具体场景按需纳入，避免过度输出。

## 核心原则

1. **程序员视角**：技术方案必须面向程序员编写，使用技术语言而非业务语言；架构图、代码示例是核心产出，而非泛泛而谈的描述
2. **图表优先**：能用图表说清楚的，不要用大段文字；流程图、时序图、架构图使用【Mermaid 语法】绘制
3. **代码说话**：对于关键实现部分，必须给出可运行的关键代码示例，而非伪代码或抽象描述；代码示例参照【代码示例规范】
4. **决策有据**：技术选型必须给出选择理由和对比分析，不能只列名字不解释为什么选它
5. **需求驱动**：技术方案的所有设计决策必须回溯到用户的需求描述，不脱离需求做过度设计
6. **弹性输出**：文档结构分为必要章节和可选章节，可选章节根据需求场景判断是否纳入，不强制输出所有章节；参照【章节选择策略】
7. **适度深挖**：如果用户需求描述不够具体，必须主动追问关键细节，参照【需求追问策略】

## 工作流程

### 第一步：收集需求

1. 接收用户描述的需求（用户想做什么东西）
2. 分析需求描述，判断信息是否足够生成技术方案；关键信息包括：
   - 产品功能范围和核心用例
   - 目标用户和使用场景
   - 已有的技术栈或技术偏好
   - 规模预期（用户量、数据量、并发量）
   - 是否有现有系统需要对接
3. 如果信息不足，使用 `AskUserQuestion` 工具按照【需求追问策略】向用户提问，收集到足够信息后进入下一步

### 第二步：架构设计

1. 根据需求识别系统的核心模块和边界
2. 确定整体架构风格（单体/微服务/Serverless 等）
3. 使用 Mermaid 绘制【架构图】和【流程图/时序图】
4. 确定模块间的通信方式和数据流向

### 第三步：技术选型

1. 根据架构设计和需求特征，列出候选技术栈
2. 对关键技术选型给出对比分析和选择理由
3. 明确各层（前端/后端/数据库/基础设施）的技术选择

### 第四步：判断可选章节

1. 根据需求场景，按照【章节选择策略】判断需要纳入哪些可选章节
2. 如果需要数据模型设计，进入数据模型设计步骤
3. 如果需要 API 接口设计，进入 API 接口设计步骤
4. 如果两者都不需要，直接进入关键代码示例步骤

### 第五步：数据模型设计（可选）

1. 识别核心实体和关系
2. 设计数据模型（表结构/文档结构）
3. 使用 Mermaid 绘制 ER 图或类图
4. 说明关键索引和查询优化策略

### 第六步：API 接口设计（可选）

1. 设计核心 API 端点
2. 定义请求/响应格式
3. 说明认证鉴权方案
4. 标注关键接口的幂等性和并发处理策略

### 第七步：关键代码示例

1. 识别方案中最关键、最复杂的实现部分
2. 按照【代码示例规范】编写关键代码示例
3. 代码示例应覆盖：核心业务逻辑、复杂算法、并发处理、数据转换等关键点

### 第八步：输出文档

1. 按照【输出内容结构】组织完整文档，仅包含必要章节和已确定的可选章节
2. 自检文档完整性和一致性
3. 保存到指定位置

## Mermaid 语法

所有图表使用 Mermaid 语法编写，嵌入 Markdown 代码块中，格式为：

````markdown
```mermaid
图表类型
  图表内容
```
````

常用图表类型：
- **flowchart**：用于绘制架构图、模块关系图、业务流程图
- **sequenceDiagram**：用于绘制时序图，展示模块间交互顺序
- **erDiagram**：用于绘制 ER 图，展示数据实体关系
- **classDiagram**：用于绘制类图，展示代码结构

图表命名规范：
- 架构图命名为"系统架构图"
- 流程图命名为"xxx流程图"
- 时序图命名为"xxx时序图"
- ER 图命名为"数据模型 ER 图"

## 代码示例规范

关键代码示例必须遵循以下规范：

1. **语言明确**：每个代码块必须标注编程语言
2. **可运行**：代码必须是可运行的，不是伪代码；允许省略 import 语句但需在注释中说明
3. **聚焦关键**：只展示关键实现逻辑，用 `// ...` 省略样板代码和无关细节
4. **带注释**：在关键逻辑处添加行内注释，解释"为什么这样做"而非"做了什么"
5. **完整上下文**：代码示例前用 1-2 句话说明这段代码在方案中的位置和作用
6. **典型场景**：优先展示最典型、最复杂的场景代码，而非简单的 CRUD

## 需求追问策略

当用户需求描述不够具体时，按以下维度追问（每次最多问 3 个问题，避免用户负担过重）：

1. **功能边界**：核心功能有哪些？哪些是 MVP 必须，哪些可以后续迭代？
2. **技术偏好**：是否有指定的技术栈、语言、框架？是否有现有系统需要兼容？
3. **规模预期**：预期的用户量、数据量、并发量级？这直接影响架构决策
4. **非功能需求**：对性能、可用性、安全性的特殊要求？
5. **集成需求**：需要对接哪些外部系统或服务？

## 章节选择策略

技术方案文档分为【必要章节】和【可选章节】。必要章节始终包含，可选章节根据需求场景判断是否纳入。

### 必要章节（始终包含）

| 章节 | 说明 |
|------|------|
| 简介 | 项目背景、核心功能、适用范围 |
| 整体架构与流程 | 架构图、流程图、时序图（Mermaid） |
| 技术选型 | 技术栈总览、选型对比与理由 |
| 关键代码示例 | 核心实现的关键代码 |

### 可选章节（按需纳入）

| 章节 | 纳入条件 |
|------|----------|
| 数据模型 | 需求涉及数据持久化、有核心实体关系需要设计 |
| API 接口设计 | 需求涉及对外提供接口、前后端分离架构、第三方集成 |

### 判断规则

1. 如果需求涉及数据库/存储设计，纳入「数据模型」章节
2. 如果需求涉及 HTTP API / RPC 接口 / 前后端交互，纳入「API 接口设计」章节
3. 如果两个条件都满足，两个章节都纳入
4. 如果都不满足，只输出必要章节
5. 可选章节的编号顺延必要章节，保持文档编号连续

## 输出

- **格式**：Markdown
- **命名**：`{项目名称}技术方案.md`
- **位置**：当前项目 `docs` 目录下
- **内容结构**：

```
# {项目名称} 技术方案

## 1. 简介（必要）
  - 项目背景与目标
  - 核心功能概述
  - 方案适用范围

## 2. 整体架构与流程（必要）
  - 系统架构图（Mermaid flowchart）
  - 核心业务流程图（Mermaid flowchart）
  - 关键交互时序图（Mermaid sequenceDiagram）

## 3. 技术选型（必要）
  - 技术栈总览
  - 关键选型对比与理由

## 4. 数据模型（可选）
  - ER 图（Mermaid erDiagram）
  - 核心表/文档结构定义
  - 索引与查询优化策略

## 5. API 接口设计（可选）
  - 接口总览
  - 核心接口定义（方法、路径、请求体、响应体）
  - 认证鉴权方案

## 6. 关键代码示例（必要）
  - 各关键实现的代码示例（带注释和上下文说明）
```

注：可选章节根据【章节选择策略】判断是否纳入，未纳入的章节不输出，编号顺延。

示例参考 [example-tech-solution.md](./examples/example-tech-solution.md)

## 绝对不能做的事

1. 不要在需求不明确时强行生成方案，必须先追问关键细节
2. 不要用大段文字描述可以用图表表达的内容，图表优先
3. 不要给出无法运行的伪代码作为代码示例
4. 不要只列技术名词不给选型理由
5. 不要做脱离用户需求的过度设计
6. 不要强制输出所有可选章节，必须根据场景判断是否纳入
7. 不要输出用户未要求且场景不需要的章节
