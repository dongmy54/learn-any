根据用户提供的文本描述，生成符合标准格式的 Claude Code 技能（Skill），确保技能条理清晰、职责明确，AI 可无歧义地理解并执行。

## 核心原则

1. **理解优先**：在生成技能前，必须充分理解用户的意图和需求；如果描述不够清楚，必须主动向用户询问，使用 `AskUserQuestion` 工具提供选项让用户选择，用户也可输入自定义内容
2. **结构规范**：生成的技能必须严格遵循【技能标准结构】，确保每个模块完整、清晰
3. **职责明确**：技能中的每一条指令都必须无歧义，AI 读后应能准确理解"做什么"和"怎么做"
4. **适度裁剪**：不是所有模块都必须存在，根据技能的实际使用场景和用户描述，判断哪些模块需要、哪些可以省略；但核心模块（frontmatter、概述）不可省略
5. **示例分离**：技能中的示例文件应单独存放于 `examples/` 目录下，在主 SKILL.md 中以相对路径引用，不要将大段示例内嵌于主文件
6. **引用机制**：在文档中使用【xx】标记进行内部引用，然后在适当位置对【xx】进行详细说明，以保证行文逻辑顺畅

## 技能标准结构

一个标准技能由以下模块组成，各模块说明如下：

### ① Frontmatter（必须）

位于文件最顶部，使用 `---` 包裹的 YAML 格式：

```yaml
---
name: <英文名称>
description: <中文描述：用途 + 自动触发场景>
---
```

- **name**：技能的英文标识名，使用小写字母和连字符（如 `quick-overview`、`concept-extract`）
- **description**：用中文简要描述技能的用途以及自动触发场景，让 AI 知道何时应调用此技能

可选字段：
- **context**：如 `fork`，表示技能执行时的上下文模式
- **allowed-tools**：限制技能可使用的工具，如 `Bash(npx agent-browser:*)`

### ② 概述（必须）

紧跟 frontmatter 之后，用一段话（1-3 句）概括技能的核心目标和定位。这是技能的"一句话介绍"，让 AI 立刻理解这个技能是做什么的。

### ③ 工作流程（按需）

描述技能执行的步骤流程，使用有序层级结构：

```markdown
## 工作流程

### 第一步：xxx
1. 具体操作说明
2. 具体操作说明

### 第二步：xxx
1. 具体操作说明
2. 具体操作说明
```

- 工作流程不是必须的，根据技能复杂度判断是否需要
- 如果添加，必须保证步骤清晰、逻辑连贯，让 AI 能按步骤执行
- 每个大步骤下可以有小的子步骤
- 步骤中可使用【xx】引用，后续对引用进行说明

### ④ 核心原则（按需）

以编号列表形式列出技能应遵循的核心职责，格式为：

```markdown
## 核心原则

1. **关键字**：详细描述这条核心职责
2. **关键字**：详细描述这条核心职责
3. **关键字**：详细描述这条核心职责
```

- 每条以编号开头
- 关键字加粗，后跟"："，再接详细描述
- 核心原则用于约束 AI 的行为边界和优先级

### ⑤ 输出模块（按需）

描述技能生成的输出内容，包括：

```markdown
## 输出

- **格式**：输出的文件格式（如 markdown）
- **命名**：文件命名规则（如 `{主题}概念提取.md`）
- **位置**：文件存储位置（如项目 `tmp` 目录下）
- **内容结构**：输出内容应遵循的结构要求
```

如有示例，在输出模块中以相对路径引用：

```markdown
示例参考 [example.md](./examples/example.md)
```

### ⑥ 绝对不能做的事（按需）

约束 AI 的行为边界，防止越界操作：

```markdown
## 绝对不能做的事

1. 不要 xxx
2. 不要 xxx
3. 不要 xxx
```

### ⑦ 其他模块（按需）

根据技能的具体场景，可自行添加模块，例如：
- **交互指令**：说明用户如何触发技能
- **最佳实践**：提供使用建议
- **角色设定**：为 AI 设定特定角色
- **检查阶段**：对输出结果进行自检

添加模块的唯一标准：**是否能让 AI 更好地理解和执行这个技能**。

## 工作流程

### 第一步：收集用户需求

1. 接收用户提供的文本描述
2. 分析描述内容，判断是否包含足够的信息来生成技能，需要确认的关键信息包括：
   - 技能的用途和目标
   - 技能的触发场景
   - 技能的输入和期望输出
   - 技能执行中是否有特殊约束或边界
3. 如果信息不足，使用 `AskUserQuestion` 工具向用户提问，提供选项让用户选择，用户也可输入自定义内容；必须收集到足够信息后才能进入下一步

### 第二步：设计技能结构

1. 根据用户需求，确定技能需要包含哪些模块（参照【技能标准结构】）
2. 确定技能的英文名称（name）和描述（description）
3. 规划工作流程的步骤（如果需要）
4. 确定核心原则（如果需要）
5. 确定输出格式、命名和位置（如果需要）
6. 确定禁止事项（如果需要）
7. 判断是否需要额外的自定义模块

### 第三步：生成技能文件

1. 在 `.claude/skills/<技能英文名>/` 目录下创建 `SKILL.md` 主文件
2. 按照【技能标准结构】中确定的模块，依次编写各部分内容
3. 如果技能包含示例，在 `examples/` 目录下创建示例文件，在 SKILL.md 中以相对路径引用
4. 确保文档中使用【xx】引用机制保持行文逻辑顺畅

### 第四步：自检

1. 检查生成的技能是否完整覆盖了用户的需求
2. 检查技能结构是否符合【技能标准结构】
3. 检查工作流程是否清晰、步骤是否可执行
4. 检查核心原则是否完整、关键字是否加粗
5. 检查输出模块是否明确（格式、命名、位置）
6. 检查示例文件是否已分离并在主文件中引用
7. 如果发现问题，修正后再次检查

## 输出

- **格式**：Markdown 文件
- **主文件**：`.claude/skills/<技能英文名>/SKILL.md`
- **示例文件**：`.claude/skills/<技能英文名>/examples/` 目录下
- **内容结构**：严格遵循【技能标准结构】

完整示例参考 [example-skill.md](./examples/example-skill.md)

## 绝对不能做的事

1. 不要在用户需求不明确时强行生成技能，必须先询问
2. 不要将大段示例内容直接内嵌于 SKILL.md 主文件中，示例应分离到 `examples/` 目录
3. 不要省略 frontmatter 中的 `name` 和 `description` 字段
4. 不要生成与用户描述无关或超出用户意图的模块
5. 不要使用模糊、有歧义的表述编写工作流程和核心原则
